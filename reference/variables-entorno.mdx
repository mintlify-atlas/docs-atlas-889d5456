## Variables disponibles

<ParamField query="LEFA_VENV" type="string" optional>
  Ruta absoluta al directorio del entorno virtual. Cuando no estĂˇ definida, `run_app.py` aplica la siguiente lĂłgica de fallback:

  1. Si el directorio del proyecto (`PROJECT_DIR`) **es escribible**, usa `PROJECT_DIR/.venv` (modo desarrollo).
  2. Si el directorio del proyecto **no es escribible** (instalaciĂłn en `/usr/share/â€¦`), usa `$XDG_DATA_HOME/lefa/.venv` (o `~/.local/share/lefa/.venv` si `XDG_DATA_HOME` no estĂˇ definida).

  ```bash

  ```
</ParamField>

<ParamField query="LEFA_PROJECT_DIR" type="string">
  Variable **de solo salida**: `run_app.py` la fija automĂˇticamente con el valor de `PROJECT_DIR` (siempre `Path(__file__).resolve().parent`) y la propaga al proceso hijo mediante `os.environ.setdefault`.   Su utilidad es informativa: permite que scripts externos o herramientas de CI/CD lean `LEFA_PROJECT_DIR` para conocer el directorio del proyecto sin necesidad de calcularlo por su cuenta.

  ```bash
  # Solo lectura desde fuera de LEFA; no es necesario exportarla manualmente
  echo $LEFA_PROJECT_DIR
  ```
</ParamField>

### LĂłgica completa en `run_app.py`

El fragmento siguiente muestra exactamente cĂłmo `get_venv_dir()` resuelve la ruta al entorno virtual:

```python

    Directorio del entorno virtual.

    - Desarrollo (Ăˇrbol editable): PROJECT_DIR/.venv

    - LEFA_VENV: ruta explĂ­cita
    """

    if override:
        return Path(override)
    local = PROJECT_DIR / ".venv"
    if os.access(PROJECT_DIR, os.W_OK):
        return local
    data_home = Path(
        os.environ.get("XDG_DATA_HOME", Path.home() / ".local" / "share")
    )
    return data_home / "lefa" / ".venv"
```

Y asĂ­ es como `main()` fija `LEFA_PROJECT_DIR` antes de relanzar el proceso con el Python del venv:

```python

    os.environ.setdefault("LEFA_PROJECT_DIR", str(PROJECT_DIR))
    if str(PROJECT_DIR) not in sys.path:

    ...
```

`PROJECT_DIR` se calcula siempre como `Path(__file__).resolve().parent` (lĂ­nea 9 de `run_app.py`) y no puede sobreescribirse mediante una variable de entorno.

## Uso en entornos empaquetados

En una instalaciĂłn estĂˇndar (clonar el repositorio y ejecutar `run_app.py`), estas variables no requieren configuraciĂłn manual. Los casos de uso tĂ­picos donde `LEFA_VENV` resulta Ăştil son:

- **Paquetes `.deb` o instalaciones en `/usr/share/`**: el proyecto se despliega en un directorio del sistema (no escribible por el usuario), por lo que el entorno virtual debe residir en el directorio de datos del usuario (`~/.local/share/lefa/.venv`). LEFA lo detecta automĂˇticamente, pero `LEFA_VENV` permite forzar una ubicaciĂłn diferente si la polĂ­tica del sistema lo requiere.
- **Rutas personalizadas en un servidor de escritorio compartido**: cuando varios usuarios comparten una instalaciĂłn Ăşnica de LEFA en `/opt/lefa` pero cada uno necesita su propio entorno virtual, `LEFA_VENV` permite apuntar a un venv por usuario sin modificar el proyecto.
- **Entornos de CI/CD o pruebas automatizadas**: fijar `LEFA_VENV` garantiza que los scripts de prueba usen un entorno virtual conocido independientemente del directorio de trabajo actual.

<Note>
  Si `LEFA_PROJECT_DIR` no es escribible y `XDG_DATA_HOME` tampoco estĂˇ definida, el entorno virtual se crearĂˇ en `~/.local/share/lefa/.venv`. AsegĂşrese de que el usuario tenga permisos de escritura en `~/.local/share` antes del primer arranque.
</Note>
